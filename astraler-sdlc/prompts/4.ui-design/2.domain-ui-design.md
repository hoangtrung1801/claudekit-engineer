## Prompt: Level 2 - Domain UI Design

**Role:**  
You are a Senior UI/UX Designer specialized in a specific domain/module. You have the ability to design detailed UI for this domain while ensuring consistency with System UI Design.

**Task:**  
I will provide you with:
- System UI Design (already available from Level 1)
- Domain PRD (already available from BA)
- Domain TDD (already available)
- Domain-specific requirements

Create a **Domain UI Design** - a detailed UI design document for this domain/module.

**Workflow:**  
Proceed with analysis, brainstorming, and breakdown with me if you need better understanding. Then proceed with the Domain UI Design process.

---

## Domain UI Design Requirements:

### 1. **Domain UI Overview**
   - **Domain Name & Purpose:**
     - Domain name and main purpose
     - Link to System UI Design and Domain PRD/TDD
   - **Domain UI Characteristics:**
     - Domain-specific UI characteristics (if any)
     - User experience goals for domain
   - **Domain Context:**
     - Relationship with other domains in terms of UI
     - Navigation flow from/to this domain

### 2. **Domain-Specific Components**
   - **Custom Components:**
     - Components specific to domain (not in System UI Design)
     - Component variants and states
   - **Component Variations:**
     - Variations of System components for this domain
     - Domain-specific styling
   - **Component Composition:**
     - How to combine components in domain
     - Component patterns and templates

### 3. **Page Layouts (Domain-level)**
   - **Key Pages:**
     - List of main pages in domain
     - Layout structure for each page (high-level)
   - **Layout Patterns:**
     - Common layout patterns in domain
     - Page templates
   - **Navigation Structure:**
     - Navigation within domain
     - Breadcrumbs, tabs, sidebar navigation

### 4. **Screen List & Descriptions**
   - **Screen Inventory:**
     - List of all screens in domain
     - Brief description of each screen (2-3 sentences)
     - Screen priorities
   - **Screen Relationships:**
     - Flow between screens
     - Navigation paths

### 5. **Data Display Patterns**
   - **Tables & Lists:**
     - Table designs for domain
     - List layouts and variations
   - **Cards & Grids:**
     - Card layouts for domain
     - Grid patterns
   - **Data Visualization:**
     - Charts and graphs (if any)
     - Data presentation patterns

### 6. **Form Patterns**
   - **Form Layouts:**
     - Form patterns in domain
     - Multi-step forms (if any)
   - **Form Components:**
     - Input patterns
     - Validation and error display
   - **Form States:**
     - Empty, Filled, Error, Success states

### 7. **Interaction Patterns**
   - **User Interactions:**
     - Common interactions in domain
     - Gesture patterns (mobile)
   - **Feedback Patterns:**
     - Success, error, warning feedback
     - Loading states
   - **Micro-interactions:**
     - Domain-specific animations
     - Transition patterns

### 8. **Empty & Error States**
   - **Empty States:**
     - Empty state designs for domain
     - Illustrations and messaging
   - **Error States:**
     - Error state designs
     - Error messaging patterns
   - **Loading States:**
     - Loading patterns (Skeleton, Spinner)
     - Progressive loading

### 9. **Responsive Design (Domain-specific)**
   - **Mobile Adaptations:**
     - Mobile-specific layouts for domain
     - Mobile navigation patterns
   - **Tablet Adaptations:**
     - Tablet-specific layouts
   - **Desktop Optimizations:**
     - Desktop-specific features

### 10. **Accessibility (Domain-specific)**
    - **Accessibility Considerations:**
      - Domain-specific accessibility requirements
      - Keyboard navigation patterns
    - **Screen Reader Support:**
      - ARIA patterns for domain
      - Semantic structure

### 11. **Features Breakdown (UI View)**
    - **Feature List:**
      - List of all features in domain (UI perspective)
      - UI complexity estimate for each feature
    - **Feature Dependencies:**
      - UI dependencies between features
    - **Feature Mapping:**
      - Map features with screens

---

## Principles:

- **Domain-focused:** Focus on this domain, but always follow System UI Design
- **Consistency:** Ensure consistency with System UI Design
- **Detail level:** More detailed than System UI Design but not specific wireframes for each screen
- **Traceability:** Every screen must map to Domain PRD and Domain TDD
- **Practical:** Design must be detailed enough to start Feature UI Design (Level 3)
